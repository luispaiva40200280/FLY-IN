#### Option A: The State Engine (The Referee)

This module acts as the simulation's rule enforcer and turn tracker.

- **Responsibilities:** It will instantiate the `Drone` objects (D1 through DN). It requires an `execute_turn()` loop that validates proposed moves against the map's capacity constraints (ensuring two drones do not crash in a connection, or overfill a zone). It must also enforce the strict rule that entering a `RESTRICTED` zone forces a drone to occupy the connection for one additional turn. Finally, it must generate the formatted terminal output (e.g., `D1-roof1 D2-corridorA`).
    
- **Why start here?** Establishing the engine first provides a testable framework. You cannot safely test a path-finding algorithm if you do not have a rigid engine to validate whether the calculated paths actually obey the project's physics constraints.

### Part 1: The State Engine (The Physics & Referee)

The State Engine is responsible for advancing the simulation turn-by-turn and verifying that no moves violate the map's constraints. It is the gatekeeper of valid states.

**1. The "Simultaneous Move" Paradox** The specification dictates that drones moving out of a zone free up capacity for that _same turn_. This means you cannot evaluate drone moves sequentially (e.g., moving Drone 1, updating the map, then moving Drone 2). If Zone B is at maximum capacity, Drone 2 can still enter Zone B _if and only if_ Drone 1 is simultaneously leaving Zone B on that exact turn. Your engine must process moves as a unified block and evaluate the net change in zone populations.

**2. The Restricted Zone Trap** This is the most complex constraint in the engine. Moving to a `restricted` zone takes 2 turns.

- **Turn 1:** The drone leaves its current zone and occupies the _connection_.
    
- **Turn 2:** The drone MUST arrive at the restricted destination; it cannot wait on the connection for an empty space. If your engine allows a drone to enter a connection toward a restricted zone when the destination zone will be full on the arrival turn, your engine has allowed an illegal state.
    

**3. Connection Capacities** Connections possess a `max_link_capacity`. If a connection has a capacity of 1, and two drones try to swap positions (Drone 1 moves A->B, Drone 2 moves B->A) at the same time, they will collide in the connection and violate the rule.

**4. The Output Protocol**

Your engine must generate the exact required terminal output.

- Standard move: `D1-roof1`.
    
- Restricted transit move (when the drone is mid-flight on the connection): `D1-<connection_name>`.
    
- Drones that wait (stay in place) or have reached the `end_hub` are completely omitted from the output line for that turn.
    

### Part 2: The Algorithm (The Navigators)

Standard path finding algorithms like Dijkstra or A* are fundamentally insufficient for this project because they only evaluate space (X and Y). Your algorithm must evaluate **Space and Time**.

**1. Time-Space Reservation Architecture** To avoid collisions and deadlocks, your algorithm cannot simply find the shortest physical path. It must consult a "Reservation Table." If Drone 1 decides to occupy `bottleneck` on Turn 4, Drone 2's path finding must treat `bottleneck` on Turn 4 as a temporary wall (if capacity is 1). Your algorithm will essentially search a 3D grid where the Z-axis represents the simulation turn.

**2. Strategic Waiting** Your algorithm must explicitly support the action of doing nothing. If a priority path is currently saturated, it may cost fewer total turns for a drone to wait at a hub for 2 turns and then take the fast path, rather than immediately taking a massive detour. The algorithm must calculate the cost of a "wait state" versus a "move state."

**3. Cost Weighting and Zone Preferences**

The algorithm must distinguish between zone types mathematically:

- `normal`: Cost = 1.
    
- `restricted`: Cost = 2.
    
- `priority`: Cost = 1, but the specification states these _should be preferred_. This means your heuristic or edge-weighting logic should artificially lower the cost of priority zones during path calculation so the algorithm actively hunts them out, even if a normal path is exactly the same physical distance.
    
- `blocked`: Infinite cost / inaccessible.

### 1. The Fixed-Step Game Loop

The simulation proceeds in discrete turns. The engine must process a single "tick" where it reads proposed moves, evaluates them, and updates the world state.

- **The Problem:** If you process Drone 1's move, update the map, and then process Drone 2, you violate the simultaneous movement constraint. Drone 2's evaluation will be based on a future state, not the current state.
    
- **The Solution:** A Fixed-Step Engine. You collect all intended actions for $T=1$, validate them against the state of $T=0$, and then apply the valid changes simultaneously to create the state for $T=1$.
    
- **Where to Study:** * Read the **"Game Loop"** chapter in _Game Programming Patterns_ by Robert Nystrom (available free online at `gameprogrammingpatterns.com`).
    
    - Search for articles on **"Discrete-Time Simulation" (DTS)** architecture.
        

### 2. Deferred State Update (Double Buffering)

The project dictates that drones moving out of a zone free up capacity for that _same turn_.

- **The Problem:** If Zone A has a capacity of 1 and is currently occupied by Drone 1, Drone 2 wants to enter. If you check capacity immediately, `1/1` is full, and Drone 2 is rejected.
    
- **The Solution:** The Double Buffer pattern (commonly used in rendering and cellular automata like _Conway's Game of Life_). You maintain two states: `current_state` and `next_state`.
    
    1. First loop: Iterate through all drones leaving a zone. Decrement the occupancy in `next_state`.
        
    2. Second loop: Iterate through all arriving drones. Check if `next_state` has available capacity. If yes, increment it.
        
    3. Final step: Overwrite `current_state` with `next_state`.
        
- **Where to Study:** * Read the **"Double Buffer"** chapter in _Game Programming Patterns_.
    
    - Study the implementation logic of **Cellular Automata algorithms** (specifically how they handle simultaneous grid updates).
        

### 3. Finite State Machine (FSM)

Your `data.py` file already defines a `DroneState` enum (`WAITING`, `IN_TRANSIT`, `ARRIVED`).

- **The Problem:** A drone entering a `restricted` zone requires 2 turns to complete the move and occupies the connection during transit. The engine must track mid-flight drones without treating them as being inside a node.
    
- **The Solution:** Implement a Finite State Machine within the engine's update loop. If a drone's state transitions to `IN_TRANSIT`, the engine registers it on the connection's capacity instead of a zone's capacity. It then sets a counter (e.g., `turns_remaining = 1`). On the next loop, the FSM forces the drone to transition to `ARRIVED` at the destination, bypassing standard movement validation because the commitment was already made.
    
- **Where to Study:**
    
    - Read the **"State"** chapter in _Game Programming Patterns_.
        
    - Research **"Moore Machines"** and **"Mealy Machines"** for understanding how state transitions are triggered by inputs (valid moves) versus elapsed time.
        

### 4. Discrete-Event Scheduling (Future Action Queuing)

Because drones interacting with restricted zones "MUST arrive at the destination after the specified number of turns", the engine must guarantee that capacity exists in the future.

- **The Problem:** The engine must prevent Drone 2 from claiming the last spot in a restricted zone if Drone 1 is already in transit and scheduled to arrive on that exact turn.
    
- **The Solution:** An Event Queue. When Drone 1 enters transit, the engine pushes an event to a schedule: `Turn + 1: Drone 1 occupies Zone X`. During the validation phase (Phase 2) of any turn, the engine must check both the current physical occupancy AND the Event Queue to ensure incoming drones do not steal capacity from in-transit drones.
    
- **Where to Study:**
    
    - Read the documentation for Python's **SimPy** library (`simpy.readthedocs.io`). _Note: You cannot use this library for the project_, but studying its core concepts (Environments, Processes, and Events) will teach you how to write your own lightweight discrete-event scheduler from scratch.

#### Option B: Space-Time A* Algorithm (The Navigator)

This is the core algorithm that calculates the routes.

- **Responsibilities:** Standard path finding only evaluates X/Y coordinates. Space-Time path finding adds a Z-axis: Time (Turns). You will need to design a `ReservationTable`—a dictionary or 3D grid mapping `(zone_name, turn_number) -> drone_count`. The algorithm must treat a node as a temporary wall if the reservation table indicates it is at maximum capacity for that specific turn.
    
- **Why start here?** It is the primary mathematical challenge of the project and directly dictates what data structures the State Engine will eventually need to process.
